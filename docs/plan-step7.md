## 详细分析

### ① 自动化风险管理 ⭐ 推荐 #1

**CEO 判断：** 当前 SC2607 持仓 SL=0 TP=0 是风险敞口。在实盘之前，必须先有风控。风控功能本身就是飞轮的一部分——更好的风控 → 敢持仓更多 → 更多收益数据 → 更好的风控模型。这满足"不变的事"：无论市场怎么变，风险管理永远是必要的。

**CTO 评估：** 复杂度 3/10。基于现有 Pipeline 架构扩展，只需：
- 在 position_tracker.py 中添加 SL/TP 价格判断逻辑
- 在 Pipeline 扫描循环中加入风控检查步骤
- 在 notification 中加入风控告警
- 无需新增依赖，SQLite 完全能支撑

**Munger 审查：** 致命缺陷 → 无。但风险是：过度自动化导致错误的止损/止盈逻辑反而亏损。缓解方案是 Paper 模式先跑验证、人工监管过渡期。

### ② 实盘 API 对接 ⭐ 备选 #2（长远必须）

**CEO 判断：** 终极目标。但现在是单向门（涉及真金白银），需要更充分的准备。建议先完成风控再实盘。

**CTO 评估：** 复杂度 8/10。需要：
- 研究和对接期货公司 API（开户、认证、交易接口）
- 从 SQLite 中读取信号并提交实盘订单
- 实盘错误处理、回滚、对账机制
- 合规和安全考虑（API Key 管理）
- 需要新增 HTTP 客户端库，可能还需要 WebSocket

**Munger 审查：** ⚠️ 潜在致命缺陷。如果风控不完善就上实盘，可能爆仓。建议推迟到 Step 7.2 或 Step 8。

### ③ Dashboard 实时化 ⭐ 不太优先

**CEO 判断：** 纯 UI 增强，不产生利润。虽然好看但不着急。但作为副线可以在风控上线后补充。

**CTO 评估：** 复杂度 5/10。需要添加 WebSocket 或 SSE 推送到前端，或定时刷新机制。当前 Flask + Jinja2 可以升级为 HTMX 或简单 JS 轮询。

**Munger 审查：** 无致命缺陷。但问题是——"这解决了什么真实问题？"当前 Dashboard 能看到数据，只是不实时。对 Paper Trading 来说这不是瓶颈。

### ④ P2 修复 ⭐ 低优先级

**CEO 判断：** 边际收益递减。Step 6 已经修了 P0/P1，P2 的 ROI 很低。

**CTO 评估：** 复杂度 1/10。但分散精力。

**Munger 审查：** 这是最安全的方向，但也是最不增值的方向。"不做 P2 修复"不会导致灾难。

## 综合推荐

```
排名 | 方向         | 理由
:---: | :----------- | :----------------------------------
🥇    | ① 自动化风控 | 最紧急 + 最简单 + 长线价值
🥈    | ③ Dashboard | 容易做、看得见的价值
🥉    | ② 实盘 API   | 终极目标但需要风控前置
🔄    | ④ P2 修复   | 有空再做
```

### CEO 最终判断
> **选择 ① 自动化风险管理。** 这是目前唯一同时满足紧急（SC2607 裸露风险）、简单（3/10 复杂度）、客户第一（阿勇需要保护资金）、长期主义（风控是所有交易系统的基石）的方向。

### Munger 复核
> 方向 ① 没有致命缺陷。警惕点：止损/止盈逻辑不要写死，保留手动 override 能力。先用 Paper 验证风控逻辑的准确性。

### CTO 确认
> 方向 ① 在当前技术栈上完全可行。建议 Pipeline 中新增风控步骤作为独立模块，与交易逻辑解耦。

### 架构设计

```
Pipeline 扫描 →
  1. market_data.py (行情获取)
  2. signal_generator.py (信号生成)
  3. position_tracker.py (持仓管理) ← 已有
  4. ⏩ risk_manager.py (风控) ← 新增
  5. notification.py (通知)
  6. dashboard update
```

`risk_manager.py` 职责：
- 读取所有 active positions
- 对每个持仓检查是否触及 SL/TP
- 如果触及 → 调用 position_tracker.auto_close_position()
- 检查保证金是否充足（追保预警）
- 输出风控摘要到通知
