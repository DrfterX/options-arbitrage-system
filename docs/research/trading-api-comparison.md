## 1. 执行摘要

当前信号系统已稳定运行，65 个期货品种（商品期货 + 股指期货），N 型结构检测 3 级评分（ENTRY/CANDIDATE/WATCH），Black-76 期权策略双引擎。需要选择一个或多个交易 API 完成从信号到实盘执行的闭环。

**核心推荐：首期 TQSdk（天勤量化），二期 vnpy（含 CTP 封装）。** TQSdk 以最低的接入成本和零资金风险完成全链路验证；验证通过后，通过 vnpy 接入 CTP 获得更低延迟和更完善的品种覆盖。

## 2. 各方案详细调研

### 2.1 CTP (Commodity Trading Platform)

#### 背景
CTP 是由上海期货信息技术有限公司（上期技术）开发的期货交易接口，是中国期货市场的事实标准。85% 以上的期货公司支持 CTP 接口，覆盖国内四大期货交易所（上期所、大商所、郑商所、中金所、广期所）。

#### 接入条件

| 项目 | 详情 |
|------|------|
| **最低资金** | 50 万元保证金账户额度（期货公司要求，非交易所要求） |
| **期货公司配合** | 必须通过期货公司营业部申请；永安期货、中信期货、南华期货等可向个人开放 |
| **申请流程** | 开户 → 营业部提交 CTP 接口申请 → 签署协议 → 获取 broker_id + 账号密码 |
| **仿真环境** | SimNow（上期技术官方仿真平台），零资金即可注册测试 |
| **完成时长** | 资料齐全情况下 3-7 个工作日 |

#### 费用结构

| 项目 | 金额 |
|------|------|
| **CTP 接口费** | 免费（上期技术不收费） |
| **期货公司开户费** | 免费 |
| **交易所手续费** | 按交易所标准收取（期货公司代收） |
| **手续费返佣** | 部分期货公司对高频/大资金客户有返佣 |
| **行情数据费** | 交易所 Level-1 行情免费；Level-2 行情需额外付费（约 150-300 元/月/交易所） |

#### API 技术质量

| 维度 | 评估 |
|------|------|
| **协议** | TCP 二进制（FIX 风格），非 HTTP/WebSocket |
| **SDK 形态** | C++ DLL（Windows）、lib（Linux） |
| **Python 封装** | vnpy 内置封装最成熟；独立封装有 [ctp-python](https://github.com/SwanLab-AI/ctp-python)（SwanLab 维护，约 2k stars） |
| **文档质量** | 上期技术官方文档齐全但 C++ 原生；Python 封装文档依赖第三方 |
| **稳定性** | 行业最高 — 期货公司生产环境长期运行无故障 |
| **延迟** | 毫秒级（TCP 直连前置机，内部撮合） |
| **行情** | 支持 Level-1 和 Level-2 行情 |
| **交易** | 下单、撤单、查询持仓/资金/成交/挂单完整覆盖 |
| **期权** | 支持商品期权交易（但期权组合保证金需额外申请） |

#### 期货品种覆盖

| 交易所 | 品种 | CTP 支持 |
|--------|------|:--------:|
| 上期所 (SHFE) | 铜、铝、锌、铅、镍、锡、黄金、白银、螺纹钢、线材、热卷、橡胶、燃料油、沥青、纸浆、不锈钢、原油期权 | 全支持 |
| 大商所 (DCE) | 豆粕、豆油、大豆、玉米、玉米淀粉、鸡蛋、生猪、棕榈油、聚乙烯（LLDPE）、PVC、聚丙烯（PP）、乙二醇、苯乙烯、焦炭、焦煤、铁矿石、纤维板、胶合板、液化石油气 | 全支持 |
| 郑商所 (CZCE) | 棉花、棉纱、白糖、苹果、红枣、尿素、纯碱、玻璃、甲醇、PTA、短纤、动力煤(暂停)、硅铁、锰硅、锰硅、菜粕、菜油、花生、对二甲苯、烧碱 | 全支持 |
| 中金所 (CFFEX) | 沪深300、上证50、中证500、中证1000（股指期货+股指期权）、国债期货 | 全支持 |
| 广期所 (GFEX) | 工业硅、碳酸锂 | 全支持 |

#### 社区和技术支持

- 上期技术官方技术论坛 (SimNow 论坛)
- vnpy 社区大量 CTP 对接帖子
- QuantConnect 等国际平台支持 CTP 数据
- 期货公司营业部提供基础技术支持

#### 与当前系统集成难度：高

| 难点 | 原因 |
|------|------|
| CTP 是 C++ SDK，Python 调用需要 ctypes 封装或通过 vnpy |
| TCP 直连前置机需要穿透 NAT 或使用专线 |
| 认证流程复杂（AppID + AuthCode + BrokerID + UserID + Password） |
| 多线程架构（行情线程和交易线程分离） |
| 50 万资金门槛对独立开发者偏高 |

### 2.2 掘金量化 (Goldmine / MyQuant)

#### 背景
掘金量化是恒生电子（中国金融 IT 巨头）旗下的量化交易平台。提供 Python SDK，覆盖股票、期货、期权。平台模式（用户策略在掘金服务器或本地运行，通过掘金 API 交易）。

#### 接入条件

| 项目 | 详情 |
|------|------|
| **最低资金** | 无硬性要求（期货账户开通即可） |
| **期货公司配合** | 需要开通支持掘金的期货账户；部分期货公司（如中信期货）有合作 |
| **申请流程** | 注册掘金账号 → 策略开发（模拟盘）→ 绑定期货资金账号 → 实盘 |
| **模拟盘** | 免费、无限额 |
| **完成时长** | 注册即用，绑定期货账号需 1-3 天 |

#### 费用结构

| 版本 | 价格 | 限制 |
|------|------|------|
| **免费版** | 0 元 / 年 | 限 2 个策略同时运行 |
| **标准版** | 5,888 元 / 年 | 无限策略、机构功能 |
| **机构版** | 定制报价 | 专业功能 |

#### API 技术质量

| 维度 | 评估 |
|------|------|
| **协议** | REST + WebSocket（内部封装） |
| **SDK 形态** | Python SDK（`pip install myquant` 或通过掘金客户端） |
| **Python 封装** | 官方提供完整 Python SDK |
| **文档质量** | 官方文档齐全，示例丰富 |
| **稳定性** | 恒生电子背书，生产级可靠性 |
| **延迟** | 中等（服务器转发模式，非直连交易所） |
| **行情** | 全品种 Level-1/Level-2 行情 |
| **回测** | 内置回测引擎，支持事件驱动回测 |
| **事件驱动** | 内置 Event Engine，策略订阅行情/成交事件 |

#### 品种覆盖

| 市场 | 覆盖 |
|------|:----:|
| 国内期货 | 全品种（上期所、大商所、郑商所、中金所、广期所） |
| 股票 | 沪深 A 股全市场 |
| 期权 | 商品期权 + 股票期权（50ETF、300ETF 等） |
| 期货期权 | 支持 |

#### 社区和技术支持

- 恒生电子官方技术团队支持
- 官方论坛 + 文档
- 企业版有专属技术支持

#### 与当前系统集成难度：中

| 难点 | 原因 |
|------|------|
| 掘金 API 与当前 AKShare 数据管线独立，需在交易层适配 |
| 掘金的事件驱动架构与当前轮询架构理念不同 |
| 免费版限 2 个策略，对 65 品种筛选型策略可能不够 |
| 掘金 SDK 加载较重量级（内置事件引擎和数据库） |

### 2.3 天勤量化 (TQSdk)

#### 背景
天勤量化是上海信易信息科技股份有限公司（Shinny Technologies）旗下的量化交易平台。核心优势是 `pip install tqsdk` 即用的极简接入体验，以及国内量化社区中良好的口碑。

#### 接入条件

| 项目 | 详情 |
|------|------|
| **最低资金** | 无（实盘账户开通即可，无额外资金门槛） |
| **期货公司配合** | 天勤自身不是期货公司，需要已有期货账户（支持 CTP 的期货公司均可） |
| **申请流程** | `pip install tqsdk` → 注册天勤账号 → 在快期客户端绑定期货资金账号 → 通过 TQSdk 连接实盘 |
| **模拟盘** | 免费，无需期货账号，注册即用 |
| **完成时长** | 模拟盘：10 分钟；实盘：开通期货账户 + 快期绑定 1-3 天 |

#### 费用结构

| 版本 | 价格 | 说明 |
|------|------|------|
| **TQSdk 社区版** | 免费 | 基础行情 + 交易功能，功能完整 |
| **TQSdk 专业版** | 约 1,000-3,000 元/年（具体价格以官网为准） | 高级功能、更高调用频率限制 |
| **TQSdk 企业版** | 定制报价 | 机构级功能 |

#### API 技术质量

| 维度 | 评估 |
|------|------|
| **协议** | WebSocket（实时行情）+ REST（交易查询） |
| **SDK 形态** | 纯 Python SDK，`pip install tqsdk` 即用 |
| **Python 兼容性** | Python 3.8+，支持 3.13 |
| **文档质量** | 官方文档完善（doc.shinnytech.com），示例代码丰富 |
| **稳定性** | 中等 — 天勤服务器作为中间代理，多了网络跳点 |
| **延迟** | 中等（WebSocket 延迟约 100-500ms，非直连交易所） |
| **行情** | 全品种期货 + 期权实时 Level-1 行情 |
| **历史数据** | 内置历史数据接口（K 线、Tick） |
| **交易** | 下单、撤单、持仓查询、资金查询、委托查询 |
| **回测** | 内置回测引擎，支持 K 线回测和 Tick 回测 |
| **扩展** | TA-Lib 兼容的技术指标模块；期权交易支持 |
| **外盘** | 支持外盘行情（美股、原油、外盘期货等） |
| **换月** | 自动主力合约换月处理 |

#### 品种覆盖

| 市场 | 覆盖 |
|:-----|:----:|
| 国内期货 | 全品种（上期所、大商所、郑商所、中金所、广期所） |
| 商品期权 | 全品种 |
| 股指期权 | 支持 |
| 外盘行情 | 部分指标性外盘（美豆、美原油、A50 等） |

#### 社区和技术支持

- 天勤量化官方论坛 (qa.tq18.cn)
- 官方文档（中文，详尽）
- 社区活跃，FAQ 覆盖面广
- 专业版/企业版有更快的支持通道

#### 与当前系统集成难度：低（推荐）

| 优势 | 原因 |
|------|------|
| `pip install tqsdk` 直接安装，无额外依赖冲突 |
| WebSocket 无需端口映射、无需公网 IP |
| 纯 Python，与当前 Python 3.13 + SQLite 架构完全兼容 |
| 提供模拟盘，零资金风险链接全链路 |
| 可以在当前 pip venv 中直接集成，无需额外进程 |

### 2.4 迅投 XTQuant (QMT)

#### 背景
迅投 XTQuant 是迅投科技（原迅投证券/迅投量化）旗下的个人量化交易接口，随 QMT（迅投量化终端）提供。XTQuant 是迅投向量化用户开放的 Python API，可以直接在本地 Python 环境中调用。

**关键定位：** 迅投 QMT 原本是券商客户端（类似通达信但面向量化），XTQuant 是其 Python API 库。本质上，QMT 是中国量化生态中一个重要的"券商通道"，部分券商（如国金证券）免费向客户提供。

#### 接入条件

| 项目 | 详情 |
|------|------|
| **最低资金** | 不同券商要求不同，通常 **1-50 万**不等（部分券商无资金门槛，部分要求 50 万） |
| **券商配合** | 必须通过**与迅投合作的券商**开通；国金证券、中信建投、华泰证券、招商证券等均支持 |
| **申请流程** | 开立支持 QMT 的券商账户 → 申请开通 QMT 权限 → 下载 QMT 终端 → 运行 xtquant Python SDK |
| **模拟盘** | QMT 终端自带模拟交易功能 |
| **完成时长** | 开通券商账户 1-2 天 + QMT 权限申请 1-3 天 |

#### 费用结构

| 项目 | 金额 |
|------|------|
| **XTQuant SDK** | 免费（开源，`pip install xtquant`） |
| **QMT 终端** | 免费 |
| **券商交易佣金** | 按券商标准（通常可谈到较低水平） |
| **行情费用** | QMT 内置行情（免费） |

#### API 技术质量

| 维度 | 评估 |
|------|------|
| **协议** | 迅投私有协议 + WebSocket |
| **SDK 形态** | Python SDK（`pip install xtquant`） |
| **Python 兼容性** | Python 3.7+，主要支持 Windows |
| **文档质量** | 中等 — 迅投官方有 API 参考文档，但示例不如天勤丰富 |
| **稳定性** | 高 — QMT 是券商级产品 |
| **延迟** | 较低（本地直连或券商网络环境） |
| **行情** | Level-1/Level-2 行情（取决于券商开通权限） |
| **交易** | 下单、撤单、持仓查询、资金查询、条件单、策略交易 |
| **回测** | QMT 终端内置回测引擎 |
| **多市场** | 同时支持沪深股票 + 期货 + 期权 + 转债 |

#### 品种覆盖

| 市场 | 覆盖 |
|:-----|:----:|
| 国内期货 | 全品种（与 QMT 合作的期货公司支持） |
| 股票 | 沪深 A 股（T+1 交易） |
| 期权 | 商品期权 + ETF 期权 |
| 可转债 | 支持 |

#### 社区和技术支持

- 迅投官方技术支持
- 证券营业部技术支持
- QMT 用户社区（QQ 群、知识星球等）
- 知乎/雪球上有大量 QMT 量化策略讨论

#### 与当前系统集成难度：中

| 难点 | 原因 |
|------|------|
| 需要券商账户（额外开户流程，且依赖特定券商） |
| `xtquant` 主要面向 Windows（macOS/Linux 兼容性不保证） |
| QMT 终端需要保持运行（xtquant 通过 QMT 的行情和交易通道通信） |
| 迅投文档质量不如天勤，排错需要更多时间 |
| 期货交易通过券商通道，不一定走 CTP 直连，部分高级功能受限 |

### 2.5 vnpy (VeighNa)

#### 背景
vnpy（发音同"维尼"）是国内 Python 量化领域最成熟的开源框架。最新版本 4.4.0（2026 年 5 月发布），已集成 AI 量化模块（vnpy.alpha）。它用一层统一的 Python 接口封装了底层 CTP、XTQuant、TQSdk 等多种交易 API。注意：**vnpy 是一个完整的交易框架，而非一个单独的 API**。

#### 接入条件

| 项目 | 详情 |
|------|------|
| **接入方式** | vnpy 自身不提供交易渠道，而是封装底层 API；具体的接入条件取决于底层 API（如 CTP 需 50 万，TQSdk 无门槛） |
| **安装** | `pip install vnpy` 或使用 VeighNa Studio 定制 Python 发行版 |
| **Python 版本** | 3.10+（推荐 3.13），支持 Windows / Linux / macOS |
| **数据服务** | 支持 RQData 数据服务（付费）、迅投研（付费）、TuShare（免费但有调用限制） |
| **数据库** | 支持 SQLite（默认）、MySQL、PostgreSQL、MongoDB、DolphinDB、TDengine 等 |

#### 费用结构

| 项目 | 金额 |
|------|------|
| **vnpy 框架** | 免费（MIT 开源协议） |
| **底层 API** | 取决于选择的 API（如 CTP 免费，TQSdk 免费/付费） |
| **数据服务** | RQData / 迅投研需付费（约 3,000-5,000 元/年起） |

#### API 技术质量

| 维度 | 评估 |
|------|------|
| **协议** | 统一 vnpy 接口 + 底层各种协议 |
| **SDK 形态** | Python（pip install vnpy），选择对应 gateway |
| **Python 兼容性** | 3.10+，推荐 3.13 |
| **文档质量** | 高，官方文档 + 用户手册 + 中文视频教程 |
| **稳定性** | 高 — 6 年以上生产验证，社区反馈活跃 |
| **延迟** | 取决于底层 API（通过 CTP gateway 可达毫秒级） |
| **功能** | 回测、实盘、算法交易（TWAP/Sniper/Iceberg）、价差交易、组合策略 |
| **风控** | 内置风控（交易前检查、自成交预防、委托数量限制） |
| **事件驱动** | 内置 Event Engine，异步事件驱动架构 |
| **现代特性** | vnpy.alpha 模块支持多因子 ML 策略开发（LightGBM/MLP/Lasso） |

#### 品种覆盖

| 市场 | gateway |
|:-----|:--------|
| 国内期货 | CTP / CTP Mini / 飞马 / 易盛 / TTS |
| 国内期权 | CTP（商品期权）、CTP Mini（支持所有期权） |
| 海外市场 | IB（Interactive Brokers）、易盛外盘（TAP）、直达期货 |
| 股票 | 易盛（部分）、IB 港股美股 |

#### 社区和技术支持

- GitHub 28k+ stars
- 官方文档（中文）完善
- 知乎/CSDN/博客园大量教程
- vnpy 官方论坛
- 微信交流群/QQ 群
- 企业服务团队（付费）
- AI 量化社群活跃

#### 与当前系统集成难度：高

| 难点 | 原因 |
|------|------|
| 300+ 依赖，uv 环境冲突风险高 |
| 事件驱动架构与当前轮询架构理念不同 |
| 默认 MongoDB 存储（虽可换成 SQLite） |
| 需要额外维护一个 vnpy 进程（Multi-process IPC） |
| 独立开发者排错 vnpy 内部问题时间成本高 |

**重要区分：** vnpy 优势在于它的 gateway 封装了 CTP 等底层 API——可以通过 vnpy 获得 CTP 的低延迟优势，而不必手写 C++ 封装。但作为完整框架使用时，重量架构是负担。

## 5. TOP 2 推荐方案

### 推荐一：天勤 TQSdk（首期 — 立即实施）

**评分：4.35/5 | 建议：立即实施**

#### 推荐理由

1. **最低的集成成本**：`pip install tqsdk` 直接在当前 Python 3.13 + SQLite 环境中运行。不引入任何额外依赖冲突。WebSocket 连接，无需端口映射和公网 IP。

2. **零资金风控验证**：天勤提供免费模拟盘账号，注册即用。可以在零资金投入下完成全链路验证（信号生成 → 风控检查 → 订单创建 → 模拟成交 → 仓位管理）。这是所有方案中最快的"signal-to-trade"闭环。

3. **架构完美兼容**：天勤的异步事件风格与当前轮询架构完全不冲突。可以设计为 TradingOrchestrator 定时轮询信号表 → 调用天勤 API 发单。不需要 Event Engine，不需要独立进程，不需要消息队列。

4. **充足的期货覆盖**：国内期货 + 商品期权全覆盖，恰好匹配当前系统的 65 个品种。外盘行情数据（美豆、美原油等）也可作为辅助分析。

5. **文档质量最好**：天勤的 Python SDK 文档在中文量化生态中数一数二，示例丰富。独立开发者遇到问题可以快速解决，不会卡在 API 集成上。

6. **社区版免费**：基础功能社区版完全免费，专业版性价比高（约 1,000-3,000 元/年）。

7. **模拟盘功能完整**：天勤的模拟盘基于真实行情撮合，比简单的本地仿真更真实，可以做仓位管理和滑点测试。

#### 不足

- 延迟中等（WebSocket 约 100-500ms），不适合高频交易
- 依赖天勤服务器（如果天勤服务中断，交易无法执行）
- 无法做到原生 CTP 的 Level-2 深度行情

#### 适用场景

- 信号频率低（ENTRY 信号 < 5 次/天）
- 持仓周期 1 天 ~ 1 周
- 初始资金 5,000-10,000 元
- 验证信号到交易的闭环可行性
- 极速上线（1-2 Cycles）

---

### 推荐二：vnpy（二期 — 信号验证通过后接入 CTP）

**评分：3.35/5 | 建议：信号验证后分阶段接入**

#### 推荐理由

1. **CTP 接入的最佳方式**：vnpy 的 CTP gateway 是整个生态中最成熟的 CTP Python 封装。通过 vnpy 接入 CTP，既可以享受 CTP 的毫秒级延迟和完整品种覆盖，又不需要手写 C++ CTYPES 封装。

2. **社区和生态系统**：28k+ GitHub stars，国内量化框架中使用最广泛。遇到问题几乎都能在社区找到答案。

3. **回测能力最强**：vnpy 的回测引擎是业界标准。如果需要在大规模实盘前做更严谨的回测验证，vnpy 的 CTA 回测器、组合回测器都是久经考验的。

4. **低延迟选择**：通过 CTP gateway 直连期货公司前置机，延迟从 500ms 降至 <5ms。对于需要快速反应的趋势跟踪策略，这是显著的改进。

5. **算法交易**：内置 TWAP、Sniper、Iceberg 等算法交易策略，适合需要拆单/隐蔽大单的场景。

6. **现代化**：4.4.0 版本支持 AI 量化（vnpy.alpha），虽然当前不需要，但为未来 ML 信号增强保留了路径。

#### 不足

- **当前阶段太重**：为了接入 CTP，vnp y 需要加载整个框架（300+ 依赖）。在初期资金 5000 元、每天不到 5 笔交易的情况下，这是显著过度工程。
- **架构冲突**：事件驱动 + MongoDB + 独立进程的架构与当前 SQLite 轮询架构完全不同。
- **集成需要额外进程**：作为独立进程时，信号系统和 vnpy 之间需要 IPC 通信（Redis/消息队列/socket），增加复杂度。

#### 使用策略：仅使用 vnpy 的 CTP gateway

一个更明智的策略是"取其精华"——只复用 vnpy 的 CTP gateway 部分，而不整个加载 vnpy 框架。但 vnpy 的 gateway 设计为框架的一部分运行，独立剥离需要一定工作量。**推荐等到信号系统的实盘交易价值被验证后（天勤阶段），再投入资源通过 vnpy 接入 CTP。**

#### 适用场景

- 信号质量通过实盘验证，值得扩大资金
- 延迟要求较高（持仓周期缩短到分钟级）
- 需要 Level-2 深度行情分析
- 资金规模增长到 50 万以上（CTP 门槛）
- 策略复杂度增加，需要 vnpy 的回测和算法交易

## 6. 集成路线图

### 阶段 1：天勤 TQSdk 模拟交易（1 Cycle）

**目标：完成"信号 → 风控 → 订单 → 模拟成交 → 仓位管理"全链路闭环**

| 步骤 | 任务 | 工作量 |
|:----|:-----|:------:|
| 1.1 | `pip install tqsdk` + 注册天勤账号 + 模拟盘验证 | 1 小时 |
| 1.2 | 在 `trading/` 包中实现 `BrokerTQ`（place_order / cancel / sync 等接口） | 4 小时 |
| 1.3 | 集成 TradingOrchestrator：轮询信号表 → 风控哨兵检查 → 通过 BrokerTQ 发单 | 4 小时 |
| 1.4 | 集成 PositionManager：同步天勤持仓 → 写入本地 positions 表 | 2 小时 |
| 1.5 | 集成测试：信号 → 模拟交易全链路 | 3 小时 |
| 1.6 | Web 看板：显示模拟持仓、盈亏和风控状态 | 2 小时 |

**产出：** `TradingOrchestrator` 启动后，自动从信号表读取 ENTRY，经风控检查后通过天勤模拟盘发送订单，成交后写入本地 positions 表，Web 看板实时显示持仓和盈亏。

**验证标准：** 连续运行 3 天无异常，模拟成交逻辑与天勤服务器一致。

**风险：** 天勤服务器稳定性和模拟盘撮合逻辑可能与实盘不同。如发现模拟盘与实际行情差距大，需调整模拟参数。

### 阶段 2：天勤 TQSdk 实盘验证（1 Cycle）

**目标：用最小资金（5,000-10,000 元）验证信号系统的实盘交易价值**

| 步骤 | 任务 | 工作量 |
|:----|:-----|:------:|

| 2.4 | 风控收紧：最大持仓 1 个品种、日亏损限额 200 元、信号最低评分 0.55 | 30 分钟 |

| 2.6 | 自动执行 + 每日复盘：运行 1-2 周 | 持续监控 |

**风控保障：** 实盘阶段风控哨兵全量启用（三层防御），止损优先于一切。日亏损达到 200 元（5% of 5000）自动熔断。

**实盘验证指标：**

| 指标 | 目标值 | 意义 |
|:----|:------|:----|
| 信号命中率 | > 40% | ENTRY 信号实盘盈亏为正的比例 |
| 盈亏比 | > 1.5 | 平均盈利 / 平均亏损 |
| 最大回撤 | < 10% | 账户净值的最大回撤比例 |
| 胜率 | > 35% | 盈利交易数量比例 |

### 阶段 3：通过 vnpy 接入 CTP（2 Cycles，可选）

**目标：在信号系统价值验证通过后，升级到 CTP 低延迟通道**

| 周期 | 步骤 | 任务 |
|:----|:-----|:-----|
| Cycle 3.1 | 3.1.1 | 安装 vnpy（仅安装 ctp gateway 相关依赖，可考虑轻量安装） |

| | 3.1.3 | 信号系统 ↔ vnpy 进程通信（建议使用 SQLite 共享，或简单的 HTTP RPC） |

| Cycle 3.2 | 3.2.1 | SimNow 仿真环境全链路测试 |

## 7. 关键决策依据

### 为何不选择掘金量化

掘金量化的免费版限制 2 个策略同时运行。当前系统的设计是"65 品种全量扫描，筛选 ENTRY 信号后执行"——这本质上是 1 个多品种筛选策略。但如果策略需要拆分为多个子策略（如按板块/策略参数拆分），免费版的 2 策略限制会成为瓶颈。付费版 5,888 元/年在这个阶段是不必要的成本。

### 为何不选择迅投 XTQuant

关键限制是平台：XTQuant 最好在 Windows 上运行 QMT 终端，而当前开发环境是 macOS。其次，XTQuant 通过券商 QMT 通道，期货交易的灵活性和速度不如 CTP 直连，但门槛并没有比 CTP 低太多（部分券商仍要求 50 万）。最后，xtquant 文档质量和社区支持不如天勤 TQSdk。

### 为何先 TQSdk 后 vnpy/CTP

验伪逻辑：先用最低成本验证"信号到交易"这件事是否真的能赚钱。如果信号质量不够，最应该做的是优化信号而非升级交易通道。TQSdk 的模拟盘和 5000 元实盘成本远低于 CTP 的 50 万门槛。核心原则：**不要为了一个未被证实的假设投入大量资源。**
