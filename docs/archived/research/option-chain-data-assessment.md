## 1. 实盘时段期货行情传输现状

### 当前状态
期货行情（Sina 15m K线）在**所有时段**正常返回：
- 非交易时段 (23:00+)：RB2609 最新数据 23:00:00 ✅
- 实盘验证：`futures_zh_minute_sina(RB2609, "15")` 始终返回有效数据

### 问题根因（已修复）
期权扫描失败的"无期货行情数据"消息是**连锁反应**而非数据源问题：
1. `_init_contract_cache()` 做 59×5=295 次背靠背 API 调用，无节流
2. Sina 限流 → 0/59 品种解析失败
3. `get_top40_futures()` 找不到合约映射 → 返回空列表
4. → "无期货行情数据，跳过期权扫描"

**修复效果**：基于 `call_count` 的节流策略使 59/59 品种成功解析（64 次 API 调用 ✅）

### 结论
✅ **数据源本身无问题**。实盘与非交易时段 Sina 行情均正常工作。

## 2. 非交易时段回测模式需求

### 现状分析
当前扫描管线循环执行以下步骤：
1. `data_refresh()` → 采集 K线 + MACD + 极值点 + N 型结构
2. `run_futures_scan()` → 期货信号
3. `run_options_scan()` → 期权策略

非交易时段（23:00-09:00）：
- 期货行情无新数据产生（最新数据停留在 15:00 或 23:00）
- K 线采集返回空增量 → MACD/N 型无变化 → 信号不变
- 期权链数据基于过期期货价格计算 → 无意义

### 建议方案
不添加独立的"回测模式"。改为**智能跳过逻辑**，在 orchestrator 中判断：

```
市场时间判断逻辑:
  - 日盘: 09:00-15:00 → 正常扫描（检测 15 分钟无新数据则跳过）
  - 夜盘: 21:00-23:00 → 减少频率（每 30 分钟 vs 每 15 分钟）
  - 非交易: 15:00-21:00, 23:00-09:00 → 跳过扫描
```

### 优先级
中。当前扫描在非交易时段跑完期货扫描后，期权扫描会因为无新数据而自然跳过。性能开销约 72 秒/次，非关键路径。

## 4. 15 分钟扫描频率分析

### 当前配置
- 期权扫描：`--limit 15` 品种
- 频率：launchd 每 15 分钟触发一次
- 实际耗时：~10 秒/次（15 个品种的期权链获取）

### 信号需求分析
| 策略类型 | 时间周期 | 最低信号频率 | 建议扫描频率 |
|---------|---------|-------------|-------------|
| N型期货信号 | 15m/1h/1d | 每 15 分钟 | 15 分钟 ✅ |
| 比例价差 | 持仓级（日频） | 每日 | 15 分钟过于频繁 |
| Short Strangle | 持仓级（周频） | 每日 | 15 分钟过于频繁 |
| Iron Condor | 持仓级（周频） | 每日 | 15 分钟过于频繁 |
| IV 监控 | 日频 | 每日 | 15 分钟过于频繁 |

### 优化建议
```
频率策略:
  - 09:00-15:00 (日盘): 每 15 分钟全量扫描 ✅（捕捉 N型信号）
  - 21:00-23:00 (夜盘): 每 30 分钟仅期货扫描（无期权）
  - 23:00-09:00 (非交易): 跳过扫描（或每 60 分钟仅期货数据刷新）
  - 收盘后 15:30: 一次完整期权扫描 + IV 记录（Day End 模式）
```

⚠️ 当前 launchd 每 15 分钟固定触发，无法区分时段。

### 结论
✅ **15 分钟扫描频率对 N 型期货信号是合适的**。期权策略不需要这么高频，但当前架构不影响正确性（只是多了些冗余计算）。

## 6. 发现的第二问题：SQLite 并发冲突

### 现象
全量扫描日志中出现大量：
```
unable to open database file
N型失败 XX 15m: unable to open database file
```

### 根因
`Database.get_conn()` 每次调用创建新连接（`sqlite3.connect`），61MB 数据库在 WAL 模式下被多个连接并发访问。`data_refresh()` 中 119 个品种 × 5 个时间周期 = 595 个并行的数据库操作，连接竞争导致文件锁溢出。

### 严重性
- **中高**：期货扫描在并发冲突时部分品种的 N 型检测结果为 0
- 期权扫描受期货数据依赖，间接受到影响

### 修复方向
1. 单例连接池：`Orchestrator` 持有单一连接 → 所有子函数复用
2. 增加 `timeout` 参数：`sqlite3.connect(db_path, timeout=30)`
3. WAL 模式已启用但不够 → 增加 `PRAGMA busy_timeout=5000`
4. 缩小 `data_refresh()` 中的并发度
