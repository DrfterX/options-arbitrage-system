| 条件 | 定义要求 | 算法实现 | 结论 |

### 🚩 1 小时周期标点错误的可能原因（推理）

既然算法逻辑正确，1h 浮窗标点偏差的根因可能在以下一个或多个因素：

1. **`SWING_WINDOWS["1h"] = 3` 参数不合适** — 1h K 线波动模式与 15m/3m 不同，相同的窗口 3 可能对 1h 产生过多或过少的极值点
2. **极值点质量导致算法选择了"正确但视觉不直观"的组合** — 算法按非重叠约束从最早极值点向前扫描，取最后一个有效结构。如果极值点序列中有多个可能的 ABC 组合，算法可能选了一个数学上正确但人类视觉上不认为是"当前主要结构"的组合
3. **特定品种的 K 线形态异常** — 某些品种在 1h 周期有特殊形态（如长时间横盘、跳空、流动性不足），导致 swing point 检测失效

### 排查策略（由易到难）

先验证参数敏感性 → 再定位具体品种 → 最后改参数/逻辑

## 拆解步骤

  - [ ] 调用 `_merge_same_type()` 合并

  - [ ] 显示 ABC 三点的 type/price/time
  - [ ] 显示条件 1-4 的检查结果（每个条件 true/false）

### Step 2 — 运行验证 + 记录异常（1 Cycle）
**产出物：** `docs/n-structure-verification-report.md`

- [ ] 运行脚本，收集所有品种 × 周期的结果

- [ ] 特别关注 1h 周期的异常品种
- [ ] 分析异常模式（是否集中在某类品种？同一品种在多个周期异常？）

### Step 3 — 修复（1~2 Cycles）
基于 Step 2 发现的根因选择修复方案：

- **方案 A（最可能）**：调整 `SWING_WINDOWS["1h"]` 参数（如 3→5 或 3→7），减少噪声生成的极值点
- **方案 B**：如仅个别品种异常，添加品种白名单/黑名单
- **方案 C**：如极值点本身质量差，改进 `detect_swing_points()` 逻辑（如引入价格变动百分比阈值过滤小波动）
- **方案 D**：如前向扫描选中的结构视觉上不如更近期的组合，在 `_find_n_structure_forward()` 中增加备选结构评估（如优先选择最近时间范围内的组合）

### Step 4 — 前端验证 + 部署（1 Cycle）
- [ ] 在 K 线浮窗验证修复后的 ABC 标点
- [ ] 更新共识 + 关闭任务
